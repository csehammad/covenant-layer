Covenant Network Function Plan

Planning document for how the Covenant Network functions, runs, and onboards participants.

1) Network Purpose and Operating Principle

Covenant Network coordinates outcomes across independent participants. The key boundary is:

  • Agent coordinates intent, authority, offer comparison, and acceptance.
  • Provider fulfills deterministic execution and supplies evidence.
  • Settlement service records final lifecycle state.

The protocol sits above existing HTTP request/response APIs and standardizes commitments, not internal workflows.

2) Core Actors and Responsibilities

Actor Primary Responsibilities Must Not
User Define objective, grant authority, approve when required, dispute if needed. Need to know provider internals.
Agent Interpret objective, request offers, compare, apply policy, issue acceptance. Bypass authority bounds or fabricate settlement.
Broker Route objectives, collect offers, provide traceable offer provenance. Pretend to be provider fulfillment.
Provider Issue terms, fulfill accepted commitments, submit evidence. Claim fulfillment without verifiable proof.
Evidence Service Store and expose evidence records with integrity metadata. Alter evidence payloads silently.
Settlement Service Record lifecycle state transitions and references to evidence. Settle without acceptance and evidence linkage.

3) Protocol Lifecycle (Runtime Function)

  1. Objective Created: User/agent publishes objective with constraints and preferences.
  2. Authority Checked: Grant validated for scope, budget, expiry, and approvals.
  3. Discovery + Offers: Broker routes objective to eligible providers.
  4. Comparison: Agent ranks offers and creates shortlist.
  5. Approval + Acceptance: User approval (if required), then one accepted offer.
  6. Fulfillment: Chosen provider executes via internal systems.
  7. Evidence Submission: Provider submits proof records.
  8. Settlement: Service records state (fulfilled, failed, refunded, disputed, expired).
  9. Dispute (optional): Escalation with verification and revised settlement state.

4) How the Network Runs Day-to-Day

4.1 Runtime Topology

  • Each actor runs as an independent service/process with explicit API boundaries.
  • Events are published to a network event bus and streamed via SSE for monitoring.
  • Every event includes layer, protocol_stage, and objective_id.

4.2 Operational Controls

  • Health checks for all services (/health).
  • Replay-safe IDs for objective, offer, acceptance, evidence, settlement.
  • Timestamp + expiry checks for authority and offers.
  • Audit logs for offer provenance and state transitions.

4.3 SLOs and Guardrails

  • Offer response latency budgets per objective type.
  • Acceptance-to-fulfillment timing thresholds.
  • Minimum evidence completeness requirements before settlement.
  • Automatic downgrade or throttling for low-integrity participants.

5.1 Provider Onboarding

  1. Identity: Verify operator, issue network identity, key registration.
  2. Capability Declaration: Declare supported verticals, objective classes, and constraints.
  3. Conformance Tests: Validate offer, acceptance, evidence schema compliance.
  4. Economic Terms: Configure dispute/refund policies and optional bond requirements.
  5. Probation: Limited traffic slice and stricter quality thresholds.
  6. Promotion: Move to active tier after reliability and evidence quality targets are met.

5.2 Broker Onboarding

  1. Identity + route policy disclosure.
  2. Offer provenance guarantees and anti-suppression checks.
  3. Fair routing test suite with synthetic objectives.
  4. Monitoring hooks for route transparency metrics.

5.3 Settlement Service Onboarding

  1. State machine conformance (fulfilled/failed/refunded/disputed/expired).
  2. Evidence reference integrity checks.
  3. Dispute transition correctness and immutability guarantees.
  4. Independent audit endpoint certification.

5.4 Agent Onboarding

  1. Authority handling and policy engine conformance.
  2. Approval-gating verification for purchases/commitments.
  3. Offer comparison explainability output requirements.
  4. Safety checks against unauthorized acceptance.

6) Participant Evaluation Framework

Score basis: verified outcomes, not claims.

  • Fulfillment reliability (accepted -> fulfilled)
  • Term fidelity (delivered outcome matches accepted offer)
  • Evidence integrity and timeliness
  • Dispute rate and resolution quality
  • Price stability (quote-to-final variance)
  • Latency and availability

Suggested routing policy: reputation-weighted routing with protected trial allocation for new entrants.

7) Governance and Ownership Model

  • Protocol is stewarded by open governance, not a single operator.
  • Network operation is federated: multiple brokers/providers/settlers can participate.
  • User authority remains local: users/orgs decide delegation and approval policy.
  • Upgrades are versioned: backward-compatible windows and migration periods.

8) First Implementation Milestones

  1. Ship travel vertical reference network with realtime event dashboard.
  2. Add provider probation/active/restricted state machine.
  3. Add dispute simulation path and metrics view.
  4. Publish onboarding checklists and conformance test kit.
  5. Introduce signed message envelopes for all protocol objects.

Design invariant: no settlement credit without valid acceptance and evidence linkage.
